## Section 12 — Remaining Gaps After F1–F6

### GAP A — Senior Line Fee Architecture (HIGH — root of most remaining FAILs)

**Items affected:** #14, #31, #32, #48, and cascades into #18–22, #25, #26–28, #58–59

**Root cause:** The Excel draws the full $767M facility at financial close and charges line fees on the undrawn committed balance from that point. As the facility is drawn, line fees decrease toward zero. As settlements repay the senior, line fees do NOT re-increase because the Excel does not redraw once repaid.

The app draws on-demand and repays/redraws throughout the project. This creates a cycling pattern where line fees remain high throughout because the undrawn balance oscillates.

F2 fixed the fee base from `seniorLimit` to `undrawnBalance` but could not overcome the architectural mismatch.

**Remaining fixes needed:**
1. **Option A (architectural):** Switch the senior facility to "full commitment draw" at closing, not demand drawdown. Senior is drawn $767M at snrStartIdx, revenue repays it but does not reduce the fee base (committed balance = $767M throughout).
2. **Option B (formula override):** Compute line fees on a fixed committed balance = `seniorLimit` regardless of drawn balance. This matches Excel behaviour for a committed facility.
3. **Option C (schedule):** Allow user to input a pre-agreed drawdown schedule instead of gap-fill.

**Estimated impact:** Reducing fees from $57.6M to ~$29.5M would reduce total finance costs by $28M → increase profit by $28M → app profit $142.7M + $28M = $170.7M ≈ Excel $170.1M ✅

### GAP B — PM Fee Base (MEDIUM)

**Items affected:** #13, cascades into #18–22, #25

**Root cause:** App PM fee base = ex-GST costs only (~$914M). Excel PM fee base includes GST-inclusive costs + finance costs (~$1.16B). This is a fundamental base definition difference.

**Fix needed (F7 — deferred):** Implement two-pass PM fee calculation:
1. First pass: compute PM fee on ex-GST base (current behaviour)
2. Compute GST on costs using first-pass PM fees
3. Second pass: recompute PM fee with base = costs_ex_GST + gstOnCosts + prevFinanceCosts
4. Spread second-pass PM fees into waterfall

**Estimated impact:** App $18.3M → ~$23.2M (+$4.9M costs → -$4.9M profit). After both GAP A and GAP B fixed: profit ≈ $170.7M − $4.9M = $165.8M (within 2.5% of Excel).

### GAP C — Waterfall ITC Recovery (MEDIUM — affects IRR)

**Items affected:** #22, #58–59

**Root cause:** The funding waterfall includes `gstOnCosts` as a cash outflow each period but does NOT model the corresponding ITC recovery (ATO refund). This means the waterfall needs to draw ~$72M more senior debt to cover GST payments that would otherwise be recovered. The waterfall distributes $84.6M profit while the formula shows $142.7M. IRR is based on actual waterfall distributions so shows 14.38% instead of Excel's 23.02%.

**Fix needed:** Add ITC recovery cash inflow to the waterfall. Simplest approach: each period, the bank receives `gstOnCosts[i]` back from ATO (monthly BAS cycle assumption). This nets to zero for each period, reducing the net cash deficit and therefore the required drawdowns.

```typescript
// In solveFunding, step 8 (revenue):
bankBalance += monthlyRevenue[i] - gstOnRevenue[i] + gstOnCostsRecovery[i];
// where gstOnCostsRecovery[i] = gstOnCosts[i] (or BAS quarterly timing)
```

**Estimated impact:** Once ITC recovery is modelled, waterfall profit ≈ formula profit ≈ $142.7M. IRR would improve significantly (estimate 20–23%).
